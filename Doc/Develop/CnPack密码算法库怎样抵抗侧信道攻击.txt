                   CnPack密码算法库怎样抵抗侧信道攻击
                           Revision 1.0.0.0
                       =========================
                        作者：刘啸 2026.05.26


我们在用AI审查我们的CnPack密码算法库里的安全漏洞时，发现一个共性问题：“应对侧信道攻击”能力不足。
的确，我们的密码算法库在设计之初的目的主要是正确实现加解密算法，当年几乎就没考虑过抗侧信道攻击这东西。
等到要考虑的时候，才发现，它的水还挺深。

啥是侧信道攻击？
侧信道攻击是一种不从密码算法本身的数学弱点入手，而是通过侧面分析设备（主要是CPU）在运行时的物理实现特征来获取秘密信息如密钥的攻击方式。
说人话就是，它既不反汇编你的代码，也不抓取你内存值（这两者没点权限一般都访问不着），但它会想办法捕捉分析你的代码执行时的物理信息。
啥物理信息呢？执行时间长短、功耗高低变化、电磁辐射的强弱、甚至声音、温度……
后面几个听起来太玄乎，其实它们的目的，通俗来讲，就是在不同的运行状态下分析你代码执行的时间长不长、以及累不累，从而针对性地猜测你在跑的内容究竟是什么。
为了简明起见，我们先拿“观测代码执行时间长短”这个侧信道攻击行为来举例说明。
假设我们要写一个函数比较两个动态字节数组的内容，完全一致就返回True，否则返回False，一般Delphier都会这么写：

  function CompareBytes(const A, B: TBytes): Boolean;
  var
    I: Integer;
  begin
    if Length(A) <> Length(B) then
      Exit(False);
    for I := 0 to Length(A) - 1 do
    begin
      if A[I] <> B[I] then
        Exit(False);
    end;
    Result := True;
  end;

先判断长度，不等则直接返回False，相等则挨个比较字节，只要有一个字节不同，则返回False。
不考虑CPU汇编四字节对齐批量比较之类的优化手段的话，以上写法可谓性能最优了，能够在尽可能早的时间点返回正确结果。
但这种“效率高”的写法，在密码学里，反倒成了问题所在。
什么问题？

这里假设有人能仔细地“观测代码执行时间长短”，那漏洞就浮现出来了。
该函数的典型应用场合是用来比较用户的输入和我们的某个密钥是否相同，相同则返回True，以便进行下一步开放权限的处理。
典型的加解密算法中，密钥的长度一般是已知或固定的，比如16字节。
现在有位能控制用户输入的调用者（假设他能控制字节数组A），他要通过“观测代码执行时间长短”的侧向信道攻击方式来猜出代码中用来在CompareBytes中比较的固定密钥（假设是字节数组B）。
首先他构造全0值的16个字节的字节数组A，传入，测量内部调用CompareBytes函数时的耗时（或者准确来讲是所耗的CPU时钟周期）。
全0传进去，很快就返回了，记下这个返回False所耗的时钟周期。
现在把第一个字节换成1，再传，再记。
换成2，再传，再记。
……
一般情况下，错误的输入传进去，在for循环的第一次进入时就比较不符而退出了，这个动作所耗的CPU时钟周期是固定的，假设3个周期。
但在首字节换成某个值比如42时，这个动作虽然同样返回False，但所耗的CPU时钟周期变大了（假设变成了4）。
那么就说明，这个for循环多跑了一轮。
为什么会多跑一轮？自然是第一个字节比对正确了，进一步到第二个字节错了，才如此导致的。
所以，攻击者得到了密钥的第一个字节值：42。
以此类推，每一个字节最多跑测256次就能得到正确值，整个16字节密钥，只需最多跑测4096次，便能猜出全部16个字节的准确值。
这就是侧信道攻击的威力！

那么，如何应对这种攻击呢？
有种很土的办法是在CompareBytes中加入随机时间长度的Sleep，这真能非常有效地抵御该攻击。
但这样做成本未免太高，CompareBytes本来的行为可能微秒计，这样一Sleep，变至少毫秒计了，性能损失上千倍，不宜。
正儿八经的解决方法是一种叫“固定时间干活”（Constant Time）的算法思路，在循环比对时，不匹配也不直接跳出，而是一直比到尾，最后才汇总比较结果。
上面的CompareBytes如果要写成Constant Time版，大概如下：

  function ConstTimeCompareBytes(A, B: TBytes): Boolean;
  var
    I: Integer;
    Diff: Byte;
  begin
    if Length(A) <> Length(B) then
      Exit(False);
    Diff := 0;
    for I := 0 to Length(A) - 1 do
      Diff := Diff or (A[I] xor B[I]);
    Result := Diff = 0;
  end;

这段代码的思路是，循环每个字节都比较，一次比较一对字节，只有它俩相等，其异或结果才为0，而只有所有字节对的异或结果都为0（通过or来判断），整体比较才相等。
这样，对于长度相同且固定的字节数组的比较，必须从头到尾比较完所有内容，执行时间也就固定下来，攻击者没法获得不一样的值，也就没法获得密钥。
对，顶多通过侧向信道攻击猜出密钥的正确长度。但这个不太要紧，毕竟大多数公开算法中这些长度值都是公开的。
以上ConstTimeCompareBytes函数的实现方式，也是我们CnNative中封装好的“固定时间干活”系列函数中的一个，其他固定时间干别的事儿的，还有不少。

以上例子的核心思想就是：字节比对时，持续比较、不中途跳出。
以此例子为出发点往通用、扩大化的方向总结，其实就是一句话：“机密相关的执行代码，中途不要出现跳转，确保执行动作一致”。
if啊、exit啊、continue啊、break啊，统统不要。
也许有爱思考的朋友会问，exit、continue、break这些打断了程序的正常流程，会造成代码少执行，不利于抵抗侧信道攻击，这个好理解。
但是，if then else，如果我代码块里写类似的代码，难道也不行吗？比如：

  if ACondition() then
    Result := A
  else
    Result := B;

这段代码从字面上看，无论ACondition函数返回True还是False，都会执行一个相同的给Result赋值的操作，看上去两个分支的CPU情况相同。
但别忘了，现代CPU有跳转分支预测，有指令预取，有流水线重叠执行，甚至乱序等。这样写代码，汇编中只要出现了一个JNZ（以x86汇编为例），后面的执行逻辑就不是我们能简单想象得到了。
而实施侧信道攻击的高手铁定比一般写出这类代码的程序员厉害，更精通CPU指令执行逻辑，所以为了安全起见，if也别用。
然后有朋友一定又会问：连if都不让用，我这还能好好写代码吗？
答案当然是肯定的，否则就不会出现“固定时间干活”（Constant Time）这系列编程风格了。
当然，if不是所有情况都不让用，仅只是涉及到核心机密数据如私钥、密码等处不让用，千万别扩大化了。

因为if等分支一定是以Boolean值为判断条件的，这里就首先介绍一个“固定时间干活”里的基本操作：Boolean扩展成整型。
一个Boolean值只有True和False俩值，但在Delphi领域里的计算机上实际跑的时候，它其实是“非0”和“0”两种情况。注意这个非0，它不一定是1，如果不小心由于某种机缘巧合，Boolean值出现了2啊3啊甚至255，这些值拿来做Boolean的话，都是True，只有0代表False。
基于上面这情况，我们需要将一个Boolean扩展成整型，它是False时返回全0，它是True时返回全$FF，——这操作有什么用搁后面再介绍——，具体要怎么做呢？
先看这个将Boolean扩展成八位无符号整型值的函数：

  function ConstTimeExpandBoolean8(V: Boolean): Byte;
  begin
    Result := Byte(V);
    Result := not Result;                  // 如果 V 是 True，非 0，则此步 R 非纯 $FF，R 里头有 0
    Result := Result and (Result shr 4);   // 以下一半一半地与
    Result := Result and (Result shr 2);   // 如果有一位出现 0
    Result := Result and (Result shr 1);   // 最后结果就是 00000000，否则 00000001
    Result := Result or (Result shl 1);    // True 得到 00000000，False 得到 00000001，再往高位两倍两倍地扩
    Result := Result or (Result shl 2);
    Result := Result or (Result shl 4);    // 最终全 0 或 全 1
    Result := not Result;                  // 反成全 1 或全 0
  end;

这段代码竟然出乎意料的复杂。
它的逻辑其实是，判断一个Byte值是否是0，0则返回0，非0则返回$FF。
就这个简单逻辑，竟然要写九句代码！
当然，仔细阅读的话，也不难理解。它先拿到Boolean对应的Byte数值，有八位，求反后对半对半地与下去，最后得到0或1。
当V是0时，求反得到$FF，八位全1，对半对半地与下去，最后得到1。
当V不是0时，求反得到的值里头有0，对半对半地与下去，最后得到0。
然后得到的最低位的0或1，再成倍地或上去，最终就能得到全0字节、或全1的字节。
怎么样，是不是很巧妙？全程没一个if。
类似的扩展成16位的可以这样写：

  function ConstTimeExpandBoolean16(V: Boolean): Word;
  var
    R: Byte;
  begin
    R := ConstTimeExpandBoolean8(V);
    Result := R;
    Result := (Result shl 8) or R;         // 单字节全 1 或全 0 扩成双字节
  end;

一句话，16位的拿8位扩展，32位的拿16位扩展，64位的拿32位扩展。
这样，我们就打造好了“将布尔值转换成指定字长的全$FF或全0的整型值且不用if”的基础机制。
下面要讲它的用途了。

ConstTimeExpandBoolean系列函数，它最有用的地方，就是打造一个条件交换函数系列：ConstTimeConditionalSwap，比如：

  function ConstTimeConditionalSelect8(Condition: Boolean; A, B: Byte): Byte;

它的意思是，如果Condtion是True，就把A和B两个字节的值交换，否则不交换。
其实现也很简单（比ConstTimeExpandBoolean简单多了）：

  procedure ConstTimeConditionalSwap8(CanSwap: Boolean;
    var A, B: Byte);
  var
    T, V: Byte;
  begin
    T := ConstTimeExpandBoolean8(CanSwap);
    V := (A xor B) and T;
    A := A xor V;
    B := B xor V;
  end;

如果T返回0，则V得到0（异或值被and 0给干掉了），下面三句就等同于A赋值给A，B赋值给B，没有交换动作。
但如果T返回1，则V得到A和B的异或值（被and 全1保留），这样A和该值再次异或就会得到B，B和该值再次异或就会得到A，从而实现了A、B交换。
一个if都没用，是不是很巧妙？

接下来，终于可以回到我们上面的例子了：

  if ACondition() then
    Result := A
  else
    Result := B;

这句不用if，用Constant Time风格，要如何写？答案就是上面基本技巧的组合：

  function ConstTimeConditionalSelect8(Condition: Boolean;
    A, B: Byte): Byte;
  begin
    ConstTimeConditionalSwap8(Condition, A, B);
    Result := B;
  end;

拿到Condition这个布尔变量，扩展成$FF或0，以此决定A和B是否交换，然后返回B，这就成功地以无分支的行为实现了条件选择！
这样的“分支”代码，在侧信道攻击场合看起来，行为无论如何都是相同的，毫无区别！

再补充一点，这个Condition变量的值的获取，其实也同样有Constant Time要求。
假设我们要判断俩字节值是否相等，单纯写个函数里头写Result := A = B;其实也是不符合“固定时间干活”的要求的，因为CPU很可能会将其优化成一句减法后再JNZ的跳转指令，只要有了跳转，各种副作用都会来。
所以要实现固定时间判断俩字节相等，可以这么做：

  function ConstTimeEqual8(A, B: Byte): Boolean;
  var
    R: Byte;
  begin
    R := not (A xor B);     // 异或后求反
    R := R and (R shr 4);   // 以下一半一半地与
    R := R and (R shr 2);   // 如果有一位出现 0
    R := R and (R shr 1);   // 最后结果就是 0
    Result := Boolean(R);   // 只有全 1 才是 1
  end;

它的思想和上面的扩展类似，也是判断其异或值是否是0，但不用if，而是将值一半一半地与下去，有0会得到0，全1才会得到1，再将其转换成Boolean即可。
比一句Result := A = B;虽然复杂五倍，但为了安全，还是值得的。

以上是抗侧信道攻击的一个基本原则：不走if，走Constant Time。
现在，我们拿密码算法库里椭圆曲线的标量乘法这个动作的实现来举例，进一步讲抗侧信道攻击的一些更复杂的措施。
“更复杂”的意思是，算法里耗时的、复杂的地方太多，以至于if本身带来的分支区别已经不是重点了。比如上一篇的例子，如果前后有很多复杂代码：

  // 前面很多复杂耗时代码
  if ACondition() then
    Result := A
  else
    Result := B;
  // 后面也很多复杂耗时代码

那么实际上，这个if也不太紧要，因为前后都有很多复杂耗时的代码不受这if的影响，这些复杂耗时的代码耗费的CPU时钟周期可能巨大，攻击者难以就这个if else单独实施耗时测量的攻击。
这种情况下，攻击者可能就会改变粒度，改为判断算法里有无“独立的、耗时的”块，再根据代码总体调用的耗时情况，进一步判断这种块的调用次数多少，从而获得进一步的信息。
这句话光字面上读起来不易理解，我们拿乘法的例子来说明。

等会，乘法？乘法也能侧信道攻击？不就是一条IMUL指令的事儿？
普通乘法是，可特殊的乘法就容易遭到攻击了。只是拿特殊乘法说事儿之前，我们先拿普通乘法的典型实现方式“移位相加乘法”，以一个数A乘以13这个简单的式子，先来把问题说清楚。

  Multiplier := 13;      // 乘数，二进制 1101
  Result := 0;           // 直接使用 Result 累加
  CurrentMul := A;       // 对应 2^0 倍

乘法循环如下：

  for I := 0 to 7 do
  begin
    if IsBitSet(Multiplier, I) then
      Result := Result + CurrentMul;   // 加法
    CurrentMul := CurrentMul shl 1;    // 移位
  end;

之前我们说一条IMUL指令能做完的事儿，其实内部硬件上大概也是这么实现的。
它把乘数拆成二进制表示，从其最低位朝最高位扫描，每扫一回，便把被乘数乘以二，然后这一位如果是1，表示要把被乘数加上来。所有的累加起来就是积。
在乘数是十进制13，也就是二进制1001的情况下，这个乘法的结果，就相当于A*2^3+A*2^0
如果你理解了这个乘法的实现，那么结合上面的侧信道攻击机制，你一定很快就能发现问题：
这个if IsBitSet(Multiplier, I) then，就是受攻击的根源。

如果你的乘数是二进制的1000000000000001，和乘数是二进制的1111111111111101，它的执行流程，就会有明显差别。
前者，Result:= Result + CurrentMul; 这句加法只会执行两次。
后者，则会执行十五次。
这种执行时间差别，可能就是侧信道攻击的切入点了。对方通过测量你这个乘法的执行，就能知道你的乘数里大概有多少个二进制的1，虽然还没猜出真正的乘数，但已经代表某种信息泄露了。
如果是某些功耗十分敏感的小型CPU，其加法实现和移位实现还有功耗不同的现象，那么攻击者甚至能针对性地找到乘数哪一位是1，从而恢复整个乘数。

也许爱思考的读者问，一条加法指令而已，顶多一个CPU周期，攻击者真的能测得这么准吗？
对于我们现在使用的高档CPU，这里的加法当然问题不大，可在密码学的椭圆曲线领域，那里的“加法”，问题就大了。
椭圆曲线密码学有自己一套理论，魏尔斯特拉斯曲线结合离散数学结合有限域，在上面独创了一套逻辑完整的椭圆曲线点的加法、乘法，它们的意义已经不是我们通常理解的加法和乘法了。
椭圆曲线上任意两点都能规定一个加法运算，结果是另外一个椭圆曲线上的点。
一个点加自身也行，结果仍然是另外一个点。
然后，一个点乘以一个整数（这叫做椭圆曲线点的标量乘），也是可行的，而且是椭圆曲线密码学里最常用的运算之一。
它的代码实现，不出意外的话，就是上面的移位乘。比如我们密码算法库里的乘法，原先的代码就类似于：

  R := 0; // R初始化为椭圆曲线上的零点，也用来容纳最终结果
  E := X; // E先赋值为被乘数
  
  for I := 0 to C - 1 do // C是乘数K的二进制位数
  begin
    if BigNumberIsBitSet(K, I) then
      PointAddPoint(R, E, R); // 这位是1，就把结果累加到R里
    PointAddPoint(E, E, E);   // E持续乘2的幂
  end;

而且，椭圆曲线运算的特点就是，点加运算较为耗时，因为它是定义出来的逻辑运算，并非简单的相加。
它内部有求斜率、求有限域上的逆元等各种复杂运算，比普通加法耗时多百倍，这就给侧信道攻击提供了较大的可能。
当攻击者发现椭圆曲线乘法的调用时间有明显差别的时候，就能统计出乘数里的1的多寡，甚至分析功耗，从而进一步搞事情。

写了整整三四千字，到这里才能清楚易懂地说，AI替我们找出来的问题，正是上面描述的“椭圆曲线点的标量乘法的抗侧信道攻击能力不足”。
而且，这种不足，不是if的错，是点加操作太耗时，太明显。
对应的修改思路便是，将耗时的点加操作，移到if外面，无论这位是不是1，我都实施点加。
所不同的只是，点加的结果，我根据是不是1而赋值过去，这个操作仍在if分支内，但它耗时极少，比起点加来说微不足道，综合起来足以抗侧信道攻击。
所以我们的修复方案是：

  R := 0; // R初始化为椭圆曲线上的零点，也用来容纳最终结果
  Q := 0; // 临时变量
  E := X; // E先赋值为被乘数
  
  for I := 0 to C - 1 do // C是乘数K的二进制位数，SM2中是256
  begin
    PointAddPoint(R, E, Q); // 每次都加到Q里
    if BigNumberIsBitSet(K, I) then
      R.Assign(Q); // 这位是1，就把Q加回R里，否则就抛弃Q
    PointAddPoint(E, E, E);   // E持续乘2的幂
  end;

这样的写法，针对乘数的每位二进制，都会实施PointAddPoint操作，也就是说，对于椭圆曲线里比如SM2的256位私钥，这标量乘的算法，都会点加256次，无论私钥是全0还是全1。
这样，就彻底堵死了侧信道攻击椭圆曲线标量乘的路子。

总结一下，抗侧信道攻击的核心思想始终是，让关键操作的执行路径与秘密数据无关。
无论是消除条件分支（if），还是将耗时的运算统一执行后再条件赋值，本质上都是在抹平时间与功耗的差异。
CnSM2/CnECC/Cn25519等算法中的标量乘法，经过上述改造后，即使攻击者拥有高精度的测量设备，也无法从执行时长或功耗波形中推断出私钥的任一位。
安全，便在这些看似“浪费”的固定时间操作之上而立足了。
