                   CnPack密码算法库怎样实现内存安全
                           Revision 1.0.0.0
                       =========================
                        作者：刘啸 2026.07.09



用AI持续替我们CnPack的密码算法库做安全审计，在解决掉一些漏网之鱼的小Bug，实现完抵抗侧信道攻击后，终于轮到了内存安全。
密码学相关的内存安全概念，并不是防止数组超界、访问空指针等常规的编程Bug，而是特指敏感数据应该如何及时、彻底消除，最大限度减少泄密的可能。
譬如用SM2私钥签名一段明文，我们密码库的CnSM2.pas会从PEM文件加载私钥到一个TCnSM2PrivateKey的实例中，此时该实例的一个TCnBigNumber对象会申请一片32字节长的内存区，明文持有私钥，实施签名后，该俩对象均被释放，这片内存区也被释放。
整个过程在常规编程领域看上去没啥问题，但在“内存安全”方面，就出问题了。
因为，内存被释放，并不等于数据被擦除。


大多数程序员都觉着 Free、Dispose 或 delete 之后，那块内存里的内容就不需要管了。但从安全角度来看，绝大多数内存分配器为了追求性能，只是把这片内存标记为“可以再次分配”，而不是把其中的每一个字节重新写成零。
换句话说，私钥虽然对应的对象已经不存在了，但那32字节的大整数，很可能仍然原封不动地躺在本进程的堆内存里。只要后续没有新的数据覆盖它，它就会一直保持原样。
如果此时程序因为自身或外部触发的异常崩溃生成了 Dump 文件，或者被调试器附加，又或者某个漏洞允许攻击者读取进程内存，那么这些已经“释放”的私钥仍然有机会被恢复出来。
甚至进程终止后，这片包含私钥的内存堆被释放，很可能被重新分配至其他进程，其他进程在申请虚拟内存时，也有概率触及该敏感数据。
这也是为什么密码学领域一直强调一个概念：敏感数据的生命周期，不应该随着对象生命周期结束，而应该随着数据被主动清除才真正结束。

如何安全清除内存，也是一门学问。
按AI的建议，我们只需要在TCnBigNumber的析构函数里，在FreeMemory(FD)前（FD是大数内容存储区的指针，动态申请的），将FD所指的内存区域全填0。
照理一个FillChar调用就行，但AI指出，Delphi编译器的优化手段太过诡异，很可能识别到这是一个释放内存前的标准填充动作，认为没有必要，从而优化掉。
Delphi的编译器有没有真的这么智能，还没听说过。但为了保险起见，不能用FillChar直接填，要重新写一个安全擦除内存的函数。该函数就一个要求，啥编译器都不能把它优化掉。
一开始，我们设计了一个procedure叫MemorySafeZero(Buffer: Pointer; ByteLength: Integer)，里头循环用0给指针所指的区域字节赋值。
但AI看了后，说可能不够，你写完0后至少得读一下是否为0，这样才能让编译器知道我们这个写0操作是有后续的。
于是我们又补了句读的代码，但编译器又提示，这句读的代码其实没有作用，会提示：

[dcc64 Hint] CnNative.pas(3164): H2077 Value assigned to 'VolatileSink' never used

同样可能会被编译器优化掉。
咋办呢？再进一步，将procedure改成function，将读的结果混进返回值，这下你总不能优化掉返回值了吧。
所以我们的MemorySafeZero函数的最终版是：

  function MemorySafeZero(Buffer: Pointer; ByteLength: Integer): Boolean;
  var
    P: PByte;
    I: Integer;
    VolatileSink: Byte;
  begin
    Result := False;
    if (Buffer = nil) or (ByteLength <= 0) then
      Exit;

    P := PByte(Buffer);
    for I := 0 to ByteLength - 1 do
    begin
      P^ := 0;
      Inc(P);
    end;
    VolatileSink := PByte(Buffer)^;
    Result := VolatileSink = 0;
  end;

挨个填0后，检查被填充区的第一个字节，只有是0，才返回True。
当然，这是在“不被优化”和“性能”两方面考虑后较为均衡的一个做法，如果不考虑性能，更安全的做法是反复写几次，再遍历读取数据区的每个字节，将VolatileSink去or一下，最后结果全0才算写入成功，返回True。


实现了MemorySafeZero，然后我们要做的是，考虑在啥地方调用它。
一开始为了性能，我们打算仅在一些敏感操作后，比如私钥释放前、签名过程的敏感中间变量释放前等，针对操作的大数对象调用该方法。
但后来发现，密码库实现内容众多，这样难免挂一漏万，于是干脆决定，直接在TCnBigNumber的析构方法里加入对MemorySafeZero的调用，一通百通。
这样就初步完成了CnPack密码算法库里的“内存安全”要求，也些许牺牲了一些性能。

当然，还有其他对称加解密的密钥，照理也属于内存安全编程的范围，但那些密钥管理简单，通常是用户传入内存指针，我们用指针里的内容加解密完后就返回，不存在像非对称加密那样“我们有密钥对象复制其内容并持有”的动作。所以这种情况下，内存安全的要求主要由调用者控制，我们不太适合主动做清除动作。
试想一下，调用者可能想拿一个密码循环重复解密许多内容，如果他解密一次，这个密码内容就被我们清空了，然后他第二个解密出错，这个用户铁定会给我们报Bug的。
